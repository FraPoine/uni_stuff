## Advanced Pipelining

### Increasing hardware parallelism
Usually, two complementary approaches:
- **Superpipelining**: 
	- A superpipelined processor has a has a pipeline where each logical step (IF, ID, EX, MEM, WB) is subdivided into simpler multiple pipeline stages → smaller TCLK 
- **Superscalar**:
	- superscalar architecture is a single processor that can execute several scalar operations in parallel

#### Superpipelining
- In principle, TCLK is inversely proportional to number of stages, given a total instruction latency → the simpler the stage the lower TCLK 
- However, reduction of stage complexity is limited by four factors: 
	1. ***Imbalance problems*:** Imbalance between stages reduces performance because the clock period can not be shorter than the time needed for the slowest pipeline stage 
	2. ***Pipeline overhead time***: It results from the combination of two contributions 
		- **Pipeline register delay**: this is due to setup and propagation time through the register. This is constant and related to technology 
		- **Clock skew**: worst-case delay between active clock signal edges in two points of a digital system (e.g., between any two pipeline registers) The sum of these two contributions represents lower bound to clock cycle (no time left to perform useful work) 
	3. ***Growing data dependences***: this leads to a higher number of bubbles in a deeper pipeline – performance loss due to data hazards increases 
	4. ***Longer control hazards***: imply slower branches in case of wrong predictions (i.e., wrongly chosen branches lead to flush a larger number of stages) 
- Summing up: cycle time is shorter but the number of cycles required by a given program becomes larger → usually 12-15 stages in current architectures

#### Handling hazard 
- Instructions may complete in a different order than they were issued 
- WAW hazards are now possible, since instructions no longer reach WB in order 
- WAR are still not possible, since registers are still read in ID, and instructions go through ID in order (instruction i + 1 still can’t write result before instruction i has read its operands!)
- Stalls due to RAW will be more frequent, since instructions have longer latency 
	- Must wait for the instructions to complete
